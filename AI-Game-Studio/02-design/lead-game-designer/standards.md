# Lead Game Designer — Standards

## Quality Gates (a design is "ready to build" only if…)
- It names the **pillar(s)** it serves in one sentence.
- It states the **player verb** and the **feeling** the player should have.
- It has a **failure/edge case** answer: what happens when the player ignores it, breaks it, or hates it.
- It has at least one **recorded dissent or "we considered and rejected"** ([#4](../../00-company/COMPANY.md)) — pure consensus is a red flag.
- It defines a **cheap playtest** that could prove it wrong.

## Design Review Checklist
- [ ] Maps to a pillar (or the pillar change is approved by the [Creative Director](../../01-executive/creative-director/)).
- [ ] First-10-minutes and hour-10 experience both described.
- [ ] Teachable in one screen / one tutorial beat.
- [ ] Coherent with combat / economy / systems / progression — no contradictions.
- [ ] Respects player time & money; no dark patterns ([#1](../../00-company/COMPANY.md)).
- [ ] Tier sized correctly; right council convened ([Gameplay](../../08-councils/gameplay-council/) / [Economy](../../08-councils/economy-council/)).
- [ ] Dissent recorded; memory updated.

## Common Mistakes (that this role exists to catch)
- **Feature-listing instead of designing** — shipping verbs nobody asked the pillar for.
- **Fake consensus** — a review where nobody objected because nobody played.
- **Pillar drift** — adding "just one system" until the game is about nothing.
- **Designing for the spreadsheet** — balance that's mathematically fair but feels bad.
- **Confusing complexity with depth** — depth is meaningful choice; complexity is just load.
- **Silently overruling a specialist** — kills trust and erases the reasoning.

## Balancing Guidelines (department-wide bar the Lead enforces)
Fairness is *perceived* before it is *measured* — a 50/50 fight that feels rigged is unbalanced. Tune for the median player's experience, design the ceiling for the top 5%, and never let the floor punish learning. Symmetry is a tool, not a virtue. Every balance change ships with a hypothesis and a metric.

## KPIs / Metrics
- **Pillar trace rate** — % of shipped features that map to a pillar (target: 100%).
- **First-session retention** & **tutorial completion** — does the core promise land fast?
- **Playtest fun rating** (post-session survey) trend across builds.
- **Dissent capture rate** — % of design decisions with recorded alternatives considered.
- **Re-design rate** — % of features reworked after build (lower = better up-front design).

## Best Practices
Prototype the riskiest assumption first. Keep a living "cut list" — saying no is the job. Run a playtest before every council vote when feasible. Write the pillar on the wall and read it before every review.
