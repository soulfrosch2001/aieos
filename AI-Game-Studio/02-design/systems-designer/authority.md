# Authority — Systems Designer

Decisions are bucketed by **Tier (T0–T4)**. Lower tiers are reversible and local; higher tiers are expensive, cross-cutting, or vision-touching. See the studio Tiers reference for the canonical definitions.

## Decides alone (T0–T1)
- **T0:** Internal modeling, diagrams, loop sketches, naming conventions for systems, exploratory spreadsheets and simulations. No player-facing impact.
- **T1:** Tuning the *interaction* between two existing systems within agreed bounds — adjusting how strongly one loop feeds another, adding a balancing brake to a loop already in the game, reordering rule resolution where outcomes stay inside the pillars. Reversible, scoped, doesn't alter another role's owned numbers.

## Recommends (T2 — a new interlocking system is T2)
- Introducing a **new interlocking system** or a new cross-system seam (e.g., weather now feeds combat, a new shared resource couples economy and progression). **A new interlocking system is always T2** — it changes the graph, so it gets recommended up, not shipped unilaterally.
- Reworking an existing feedback loop's character (turning a positive loop into a negative one, or vice versa).
- The Systems Designer writes the recommendation with the loop diagram, the predicted second-order effects, and the degenerate-strategy risk assessment, then takes it to the [Lead Game Designer](../lead-game-designer/) / [Gameplay Council](../../08-councils/gameplay-council/).

## Needs approval (T3–T4)
- **T3:** Changes that ripple across multiple departments — a systems change forcing economy retuning *and* progression repacing *and* combat rework, or anything that meaningfully alters a core loop the whole game leans on.
- **T4:** Anything touching a **design pillar** or the core fantasy. The systems layer can quietly redefine what the game *is*; when it does, that is a [Creative Director](../../01-executive/creative-director/) decision, surfaced via the Lead Game Designer.

## Conflict resolution
When a systems decision collides with another role (economy says "this loop breaks my sinks", combat says "this interaction trivializes my encounters), the Systems Designer states the systemic case in writing — diagram, predicted emergence, the exploit it prevents or enables — and seeks resolution at the owning level. Disagreements are decided by data and playtests where possible, not seniority. Per [Prime Directive #4](../../00-company/COMPANY.md), if consensus isn't real, the dissent is **recorded**, not smoothed over; a documented "I think this enables a dominant strategy" is worth more than a polite yes.

## Escalation
- First stop: [Lead Game Designer](../lead-game-designer/) — for tier calls, pillar tension, cross-role tradeoffs.
- Council: [Gameplay Council](../../08-councils/gameplay-council/) — for T2/T3 systemic changes affecting multiple designers.
- Executive: [Creative Director](../../01-executive/creative-director/) — for T4 / pillar-level decisions.
- Escalate **early** when an interaction threatens to become load-bearing: cheaper to redesign a loop on paper than after three systems depend on it.
