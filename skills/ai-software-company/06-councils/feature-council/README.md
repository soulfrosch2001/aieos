# Feature Council
Status: stable
Type: council
Owner: chair
Extends: none

Decides whether and how to build a feature — weighing user value against
feasibility before engineering time is committed.

## Identity
- **Convened by:** [product-manager-02](../../03-product/product-manager-02/),
  or the [CEO](../../01-executive/ceo/) when priority is contested.
- **Convened when:** T2 (new feature, cross-cutting change). See
  [engagement-tiers.md](../../00-company/engagement-tiers.md). Not standing.

## Participants
- **Chair:** [product-manager-02](../../03-product/product-manager-02/) — owns
  the value question.
- **Core:** PM, business-analyst, a lead engineer (for feasibility),
  ux-researcher.
- **Advisory (as-needed):** software-architect, designer, COO.

## Objectives
- Decide if a feature is worth building, and at what scope.
- Reconcile value with feasibility, cost, and timing.
- **Out of scope:** system structure (see
  [architecture-council](../architecture-council/)) and security risk
  acceptance (see [security-council](../security-council/)).

## Inputs
- Problem statement and the user or business need it serves.
- Evidence of value (research, data, or a stated bet).
- A rough feasibility read from the lead engineer.
