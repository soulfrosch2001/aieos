# Feature Council — Process

## Discussion Rules
Follow [../debate-protocol.md](../debate-protocol.md). Value and feasibility are
argued by different people on purpose; a feature that no one challenges has not
been examined. Dissent is recorded, never suppressed.

## Decision Process
1. The chair states the user or business problem, not the proposed solution.
2. Business-analyst and ux-researcher present the case for value.
3. The lead engineer presents feasibility, cost, and risk.
4. Council weighs value against cost and proposes a scope.
5. If consensus fails, the **product-manager-02 decides** as chair and records
   the trade-off accepted.

## Approval Gate
- **Approves alone:** building or deferring a T2 feature within current roadmap
  and budget.
- **Must escalate:** anything reshaping priorities, requiring new headcount or
  budget, or touching system structure → [CEO](../../01-executive/ceo/) for
  value; [architecture-council](../architecture-council/) for build shape.

## Escalation
- Value deadlock → CEO decides.
- Feasibility vs. value conflict →
  [Executive Orchestrator](../../01-executive/executive-orchestrator/), then CEO
  (value) or CTO (technical). See
  [../../00-company/decision-authority.md](../../00-company/decision-authority.md).
